# wiki-vs-stray-docs

## Purpose

A project that adopts a wiki and then keeps growing `docs/` folders in its main repo ends up with two source-of-truth surfaces that drift apart. Readers do not know which to trust. Maintainers update one and forget the other. The most common failure mode: a `docs/` folder revives quietly because a contributor "added a page" without knowing the project has a wiki.

This skill catches that. It does not delete anything. It does not migrate anything. It surfaces the conflict to the user and asks what they want.

## When to use

Activate when:

- A `Write` or `Edit` targets a path matching `docs/`, `doc/`, `documentation/`, `project-docs/` (default patterns; configurable).
- A new file appears in such a folder during a code review or PR scan.
- The user asks "where should this doc go?".
- The user runs `/wiki-audit` and a stray-docs folder is detected.

Skip when:

- The project has no wiki (then `docs/` is the documentation surface; this skill does not apply).
- The path matches a configured retired-folder pattern (intentional historical content).
- The path is a project-specific exception (e.g., API reference auto-generated into `docs/api/` by tooling — declare in adapter).

## Inputs

- The repo path being edited.
- The target file path.
- Detection of a sibling wiki clone (e.g., `../<repo>.wiki/`) or an in-repo wiki folder.
- The retired-folder list (from adapter cache).
- The exceptions list (from adapter cache) — paths that are legitimately auto-generated or scoped docs not in the wiki.

## Detection logic

```
1. Identify if a wiki is present:
   a. Sibling clone at `../<repo>.wiki/` (GitHub Wiki convention)
   b. Explicit `wiki/` folder inside the repo
   c. An adapter-configured wiki path
   → If none of (a), (b), (c) → SKIP this skill (no wiki exists)

2. Identify if the target path is a stray-docs path:
   - Default patterns: docs/, doc/, documentation/, project-docs/
   - Adapter-configurable to add or remove patterns
   - Path matches → candidate stray

3. Check exceptions:
   - Path matches a configured exception (e.g., docs/api/ auto-generated by tooling) → ALLOW (skip refusal)
   - Path matches a configured retired-folder → ALLOW (historical content)

4. Check intent:
   - Is the change inside an existing file (Edit)? → softer warning: "this page is in a stray docs folder; the project has a wiki; consider migrating"
   - Is the change a new file (Write)? → harder refusal: "do not create new content in a stray docs folder when the wiki exists"

5. Refuse or warn → ALWAYS surface to the user; never silently allow or silently block
```

## Decision framework

### Hard refusal (new file write)

```
REFUSED — proposed new file in stray docs folder

  Target: docs/payment-flow.md
  Wiki present: ../my-repo.wiki/ (last commit 2026-05-12)
  Stray pattern matched: docs/

  Reason: This project's wiki is the source of truth for documentation.
  Creating new content in docs/ in the main repo creates a second
  surface that will drift from the wiki.

  Options:
    A) Migrate the new content to the wiki:
       1. Create the page in the wiki:
          (cd ../my-repo.wiki && touch Payment-Flow.md)
       2. Author with the wiki-authoring template appropriate to the content.
       3. Add a link from Home / _Sidebar.
       4. Go through wiki-safe-updates to publish.

    B) Add this path to the exceptions list (rare; only for auto-generated content):
       Edit .docs-wiki.local.json under "exceptions":
       { "exceptions": ["docs/api/"] }

    C) Genuinely keep separate (rare; requires explicit reasoning):
       Reply: "keep separate: <reason>"
       The reason will be added to .docs-wiki.local.json under "knownStrays"
       so future scans do not re-flag this folder.

  Pick A, B, or C, OR provide explicit override.
```

The plugin does NOT write the file until the user resolves. It does not silently allow option A on the user's behalf.

### Soft warning (edit existing file)

```
WARNING — edit to existing stray docs file

  Target: docs/architecture.md (last modified 2025-09-04)
  Wiki present: ../my-repo.wiki/
  Stray pattern matched: docs/
  Wiki has page: Architecture.md (last modified 2026-04-12)

  Detected possible drift:
    Both this file and the wiki page Architecture.md exist.
    The wiki version is newer. Editing this file may produce
    contradictory documentation surfaces.

  Options:
    A) Migrate the edit to the wiki Architecture.md instead.
    B) Reconcile docs/architecture.md and wiki/Architecture.md
       via wiki-code-vs-docs-discrepancy.
    C) Apply this edit anyway (acknowledged drift):
       Reply: "edit stray anyway: <reason>"

  The plugin will not apply the edit until you choose.
```

The user picks. The plugin does not pick.

### Retired folder (pass through)

If the target path is inside a configured retired folder (e.g., `docs/_archived/`):

- No refusal.
- No warning.
- The plugin lets the edit proceed (subject to other skills' rules, e.g., `wiki-safe-updates` diff preview).

### Exceptions (pass through with a one-time confirmation)

If the target is under a configured exception (e.g., `docs/api/` for auto-generated content):

- Pass through.
- On first detection (per session), surface a one-time reminder: "Note: docs/api/ is configured as an exception. If this is genuinely auto-generated, fine. If a human is hand-editing here, reconsider."

## Initial scan (`/wiki-audit`)

When `/wiki-audit` runs in a project with both a wiki and a docs/ folder, this skill produces a findings section:

```
STRAY DOCS DETECTED

  Wiki present: ../my-repo.wiki/
  Stray folder: docs/
  Files in stray folder: 12

  Pages also in wiki (possible drift):
  | docs/ file | wiki page | Recommendation |
  |------------|-----------|----------------|
  | docs/architecture.md (2025-09-04) | Architecture.md (2026-04-12) | reconcile via wiki-code-vs-docs-discrepancy |
  | docs/api-reference.md (2025-12-01) | (not in wiki) | migrate to wiki OR add to exceptions list |

  Pages only in stray folder:
  | docs/ file | Last modified | Suggestion |
  |------------|---------------|------------|
  | docs/payment-flow.md | 2025-08-22 | migrate to wiki — Payment-Flow.md |
  | docs/legacy-notes.md | 2024-03-11 | likely retire — move to docs/_archived/ |

  Severity: MEDIUM (drift risk)
  Recommendation: migrate active content to wiki; archive stale content
```

The audit reports; never auto-migrates.

## Why this skill exists (the asymmetry)

A `docs/` folder in a code repo and a wiki repo have asymmetric properties:

| Property | `docs/` in code repo | Wiki repo |
|---|---|---|
| Discoverability | requires cloning the code repo; reading the file tree | URL accessible to anyone with wiki read access |
| Search | grep the repo | wiki search UI |
| Edit workflow | code-repo PR review; merged with code changes | direct edit (or PR if the project requires) |
| Version with code | yes (same commit) | no (separate repo) |
| Linkable | needs a stable URL convention | every page has a stable URL |
| Updates | tied to code releases | independent |
| Audience | code-comfortable | mixed |

Neither is wrong. The asymmetry is the point. A project that has chosen the wiki as the source of truth has chosen the wiki's properties (discoverability, mixed audience, independent updates). A growing `docs/` undermines that choice.

If a project genuinely benefits from in-repo docs alongside a wiki, the right setup is explicit: declare the exceptions list, scope the in-repo docs to the right things (e.g., README, CONTRIBUTING, an API reference auto-generated by tooling), and accept that two surfaces exist with a clear boundary.

## Safety gates

- **Never** silently allow a new file in a stray-docs folder when a wiki is present.
- **Never** silently refuse — surface the options.
- **Never** auto-migrate content from `docs/` to the wiki.
- **Never** delete a `docs/` folder.
- **Never** treat a retired folder as stray (`_archived/`, `_legacy/`, etc.).
- **Never** treat an exception path as stray (e.g., `docs/api/` auto-generated).
- **Never** assume the wiki is the source of truth in a project that has not configured one — this skill SKIPS in projects without a wiki.

## Validation checklist

Before refusing or warning:

- [ ] Wiki presence confirmed (sibling clone OR in-repo wiki OR adapter-configured path).
- [ ] Target path actually matches a stray pattern (not over-broad).
- [ ] Retired-folder list checked.
- [ ] Exceptions list checked.
- [ ] Options A / B / C surfaced (user picks).
- [ ] No file written / refused until user resolves.

## Output format

See "Hard refusal", "Soft warning", and the audit section above.

## Anti-patterns (and why)

| Anti-pattern | Why it's wrong | Correct |
|---|---|---|
| Silently refuse the write | Frustrates contributor; they bypass the plugin | Surface options |
| Silently allow the write | Drift continues invisibly | Surface options |
| Auto-migrate `docs/x.md` to wiki on first detection | Migration may be wrong (different audience, different scope) | User picks |
| Treat the README as stray | README is special; lives in repo root by convention | Default patterns do NOT match README |
| Flag a retired folder | Historical content is legitimate | Retired-folder pass-through |
| Flag auto-generated API docs | Tooling output; not human-authored drift | Exceptions list |
| Hard-refuse an edit (existing file) | The file already exists; the edit is not the original sin | Soft warning for edits; hard refusal only for new files |
| Skip the skill when both wiki and docs/ exist because "the user must know what they want" | The user may not — this skill is the moment of asking | Surface |

## Portability rationale

The skill applies to any project that has:

- A wiki (any flavour: GitHub, GitLab, Azure DevOps, MkDocs).
- A code repo where someone might create `docs/` content.

The detection logic is filesystem + adapter configuration. No platform-specific behavior.
